# Codex Agent Team Testing Design

## Goal

Validate basic workflow correctness of the `codex-agent-team` skill using real Codex collaboration in this repository, with reproducible fixture-scenario runs and verifiable evidence.

## Scope

This design covers:

- real Codex collaboration (no mocked agent responses)
- execution inside this repository (`/Users/nantasmac/projects/agentic/codex-agent-team`)
- fixture-scenario-driven validation from a dedicated `fixtures/` directory
- dual evidence collection from filesystem artifacts and session logs
- dual-layer execution entry:
  - bottom layer: harness scripts for deterministic run/collect/assert
  - top layer: skill-triggered/agent-driven entry that exercises real workflow behavior
- layered assertions:
  - existence
  - structural
  - semantic
  - must-not-happen checks

This first version does not cover:

- intelligence quality scoring (for example role quality, strategy quality, creativity)
- exhaustive robustness/recovery hardening under all failure modes

## Validation Strategy

Each fixture defines one scenario and expected outcomes. The harness executes the scenario via real Codex session flow, collects artifacts, and asserts outcomes in layers.

Recommended initial fixture set:

- `basic-happy-path`
- `role-override-path`
- `checkpoint-path`
- `recovery-prep-path`

Design intent:

- `basic-happy-path`: baseline proof that the workflow runs end-to-end with required state files and logs.
- `role-override-path`: verifies user role overrides are reflected in team/task artifacts.
- `checkpoint-path`: verifies checkpoint triggering and persisted checkpoint state.
- `recovery-prep-path`: verifies compact/recovery preparation artifacts are written before handoff/finish.

## Evidence Model

### Filesystem Evidence

Expected to include:

- run output directory per fixture (for example under `tests/out/<fixture>/`)
- copied or materialized workflow artifacts (for example `.codex/multi-agent/*` outputs)
- normalized summaries/manifests generated by harness scripts

### Session Log Evidence

Expected to include:

- raw or normalized logs/transcripts for the session used in each fixture run
- extracted markers for key phases (panel confirmation, team formation, checkpoints, recovery-prep)

A fixture is considered valid only when both evidence channels are present and mutually consistent.

## Harness Architecture

Planned scripts:

- `tests/run-fixture.sh`
  - executes one fixture scenario against real Codex workflow entry
  - writes per-run outputs and metadata
- `tests/collect-artifacts.sh`
  - gathers filesystem artifacts and session logs into a fixture-scoped output bundle
- `tests/assert-fixture.py`
  - performs layered assertions and reports pass/fail with reasons
- `tests/run-all.sh`
  - executes all fixtures sequentially and prints aggregate summary

Execution model:

1. top entry invokes the skill/agent workflow for a fixture
2. bottom harness records run context and captures evidence
3. assertion layer validates expected and forbidden outcomes

## Assertion Layers

For each fixture:

1. Existence assertions
- required files/logs/artifacts exist
- required directories and manifests exist

2. Structural assertions
- JSON/YAML/text structures are parseable and contain required keys/sections
- expected artifact schema shape is satisfied

3. Semantic assertions
- key scenario-specific expectations hold (for example role override reflected, checkpoint state updated)
- ordering-sensitive expectations hold when required (for example checkpoint before recovery-prep completion)

4. Must-not-happen checks
- forbidden outcomes are absent (for example missing panel confirmation before orchestration, contradictory state markers)
- stale or cross-fixture contamination artifacts are absent

## Scenario-to-Assertion Mapping

- `basic-happy-path`
  - must include core workflow artifacts and phase markers
  - must not show early orchestration before panel confirmation
- `role-override-path`
  - must include user override markers in logs and final team/task artifacts
  - must not silently fall back to default roles after confirmed override
- `checkpoint-path`
  - must include checkpoint trigger and persisted checkpoint records
  - must not skip checkpoint persistence at required boundary
- `recovery-prep-path`
  - must include recovery-prep artifact updates and handoff-safe state
  - must not finish without recovery-prep state update

## Repository Layout Additions

Planned additions for testing assets:

- `fixtures/`
  - one subdirectory per scenario fixture
- `tests/`
  - harness scripts and assertion program
- `tests/out/` (ignored in VCS where appropriate)
  - run artifacts and generated evidence bundles

## Success Criteria (V1)

The V1 testing system is successful when:

- all four recommended fixtures run through the harness
- each fixture produces both filesystem and session-log evidence
- layered assertions execute and report deterministic pass/fail
- failures are diagnosable from saved artifacts without re-running blindly

## Out of Scope for V1

Deferred to later iterations:

- quality-of-intelligence rubric
- stress/load/performance characterization
- full recovery robustness matrix and chaos-style interruption testing
